# AI 기반 OS 보안 점검 도구 - 종합 피드백

**평가 일시:** 2026-06-24  
**평가 모드:** 🅰 스펙 실현 가능성 평가 (팀 맥락 통합)  
**평가 대상:** MVP Spec (고도화 버전) + 팀 상황

## 📊 종합평

**🟡 조건부 가능 — SPEC의 강점은 탄탄하지만, 범위가 3주 팀 상황과 맞지 않음. "수집→분석→대시보드" 핵심 1줄기만 완주하고 나머지는 미뤄야 완성 가능**

**핵심 메시지:**
1. **스펙 설계는 우수함** (문제 정의, 사용자 페르소나, 예외 처리, AI 출력 형식이 탄탄)
2. **하지만 범위가 상용 제품 수준** (9개 대기능, 10개 성공 기준)
3. **팀은 3주 안에 핵심 3가지(수집/분석/대시보드)만 할 수 있음** (1인당 20시간)
4. **따라서 범위를 절반 이상 제거하고, 핵심 척추에만 집중해야 함**

## 📈 SPEC 평가 (구조·설계 관점)

### ✅ 강점 (살릴 것)

#### 1. 문제 정의와 사용자가 명확하고 구체적
- 「인프라(OS) 보안 점검을 수동으로 진행 중」 → 「자동화·조기 발견·대규모 관리·지속적 모니터링」
- 사용자: Admin/Operator/Viewer 역할별 RBAC까지 정리
- **의의:** 무엇을 왜 푸는지가 분명함 → 요구사항이 흔들리지 않음

#### 2. AI 출력 형식을 미리 못박음
- JSON 스키마 고정: `vulnerability_id`, `severity`, `cvss_score`, `affected_servers`, `remediation` 등
- 언어 정책: 기술 용어(영어) + 설명문(한국어)
- **의의:** AI 응답이 들쭉날쭉해지는 함정을 미리 막음 → 구현 시 시간 절약

#### 3. 실패·예외 처리를 견고하게 고려
- 「실패한 서버는 건너뛰고 나머지 스캔 계속 진행」
- 「접근 실패 시 3회 재시도, 30초 간격」
- 「CVE API 다운 시 마지막 캐시된 CVE DB 사용」
- **의의:** 운영 현실을 반영 → 프로토타입이 실제로 작동할 가능성 높음

---

### ⚠️ 약점 (제거할 것)

#### 1. 범위가 너무 넓음 (최우선 리스크)

**현황:**
- 3.0~3.8 = 9개 대기능
- 인증·암호화·아카이빙 등 횡단 기능 추가
- 성공 기준 10개 (모두 동작해야 성공으로 봄)
- → 상용 보안 제품 한 개 규모

**3주·3명(1인당 20시간)으로는 불가능:**
```
백엔드(OS 수집 + AI 분석): 20시간
프론트엔드(대시보드): 20시간
DevOps(배포·통합): 20시간
```

**결론:** 범위를 **절반 이상 제거**해야 함

#### 2. 기술 난이도가 높은 부분은 데모 범위를 좁혀야 함

**① CVE 매칭 (심장 기능)**
- 추출된 패키지 버전과 CVE 데이터베이스 자동 매칭
- 버전 비교 정확도, 매칭 로직이 까다로움
- 시간이 가장 많이 빨려들 수 있는 부분

**② Windows WinRM 수집**
- Linux SSH 한쪽만으로도 핵심 가치는 충분히 전달됨
- WinRM 테스트 환경까지 구성하는 데 오버헤드

**결론:** 
- CVE: 시연용으로 소수 패키지·소수 CVE로 좁혀서 "정확히 맞아떨어지는" 모습만 보여주기
- Windows: Linux SSH만으로 데모 → "Windows 지원"은 v1.1로 미루기

#### 3. 성공 기준 10개를 2-3개로 축소해야 함

**지금:** 시연 시나리오 1-10번까지 모두 동작해야 성공
**개선:** 핵심 1줄기만 성공 기준으로, 나머지는 보너스

## 👥 팀 상황 평가 (역량·시간 관점)

### 팀 구성
- **Claude Code + git 경험:** 1명 (병목 리스크)
- **초보자:** 2명 (DevSecOps/인프라 배경, 보안 이해도 높음)
- **작업 방식:** 비동기 + 본업 병행

### 실제 개발 시간
```
가용 시간: 3시간/일 × 20일 = 60시간
1인당: 60시간 ÷ 3명 = 20시간
```
→ 매우 짧음

### 강점
1. **DevSecOps 배경** → OS 수집·분석·취약점 이해가 깊음
2. **도커 이미지 준비** → 배포 준비 완료 (시간 절약)
3. **Claude API 할당량 확보** → 외부 의존성 해결
4. **명확한 핵심 설정** → "서버 자동 수집"이 진짜 핵심이라는 걸 알고 있음

### 약점
1. **팀원 협업 병목**
   - Claude Code+git 경험 1명이 초보 2명을 리드
   - 비동기 + 본업 병행 → 코드 리뷰 지연, PR 충돌 해결 복잡
   
2. **시간 부족**
   - 1인당 20시간은 기술 선택·셋업·개발을 동시에 하기 부족
   - 기술 스택 미결정(FastAPI/Flask, DB, 프론트엔드) → 선택 시간 추가
   
3. **학습 곡선**
   - 초보 2명이 Claude Code, git, 새 기술을 동시에 배워야 함
   - 다만 DevSecOps 배경이 있어 예외 처리·보안 이해는 쉬울 가능성

## 🎯 핵심 축 정의 (3주 완주를 위한 범위 축소)

### 꼭 필요한 것 (필수) — "척추 한 줄기"

**시연 시나리오: Linux 테스트 서버 1~5대 기준**

1. **서버 자동 수집** ✓
   - Linux SSH 기반만 (Windows 제외)
   - OS 정보, 패키지 목록, 설정값 수집
   - 소수 서버(5~10대) 병렬 처리 (기본값만, 설정 가능까지는 아님)

2. **AI 기반 분석** ✓
   - Claude API로 취약점 분석
   - CVSS 기반 심각도 판정
   - JSON 출력 (고정 스키마)
   - **CVE 매칭은 정적 테스트 데이터로** (자동 연동은 나중에)

3. **웹 대시보드** ✓
   - 메인 대시보드: 기본 KPI (서버 수, 상태, 취약점 수)
   - 취약점 목록: 심각도별 표시
   - 취약점 상세: 원인·조치방법·영향받는 서버

### 제외 항목 (나중에, v1.0 범위 밖)

- 대량 스캔 병렬화 최적화 (10-20대 동시 처리는 하되, 성능 튜닝 제외)
- 정기 스캔 자동 스케줄링 (APScheduler 통합)
- CVE 실시간 모니터링 / 자동 알림 (정적 데이터로 테스트)
- 자동 리포트 생성 (PDF/Excel 내보내기)
- 조치 추적 / 자동 재스캔
- 데이터 아카이빙 (최근 1년 DB 저장)
- Windows WinRM 수집
- LDAP/AD 연동 (로컬 계정만)

## 🛠 액션 플랜

### Phase 1: 범위 & 기술 결정 (즉시 — 내일까지)

**1-1. SPEC 범위 재정의**
- 꼭 필요한 것 3가지 문서화
- 제외 항목 명시 (이후 요청에 대비)
- 성공 기준 재작성: 10개 → 3개

**1-2. 기술 스택 최종 결정**
- **백엔드:** FastAPI (보안 기본값 우수, Python DevSecOps 팀이 빠름)
- **데이터베이스:** PostgreSQL (많이 쓰이는 표준)
- **프론트엔드:** React (초보 자료 풍부)
- **AI:** Claude API (이미 할당량 확보)

**1-3. 팀 협업 체계 정립**
- 역할 분담: 백엔드(수집+분석) / 프론트엔드(대시보드) / DevOps(통합)
- 일일 비동기 체크인 또는 주 2-3회 동기화
- 경험자 1명의 코드 리뷰/PR 통합 시간 명시 (일 1-2시간)

### Phase 2: 개발 마일스톤 (Day 1-20)

```
Day 1-2: 시스템 아키텍처 + API 설계 + DB 스키마
Day 3-5: 백엔드 기본 (FastAPI, 인증, 자격증명 암호화)
Day 6-8: OS 정보 수집 에이전트 (Linux SSH, 병렬 처리, 재시도)
Day 9-10: AI 분석 로직 (Claude API, CVSS, JSON 출력)
Day 11-13: React 대시보드 (메인, 목록, 상세, 실시간 갱신)
Day 14-16: 통합 & 테스트 (도커 배포, E2E 테스트)
Day 17-18: 버그 수정 & 성능 튜닝
Day 19-20: 시연 준비 & 최종 검수
```

### Phase 3: 시연 준비 (Day 17-20)

**시연 시나리오 (3개 성공 기준만)**

1. **서버 자동 수집**
   - 테스트 Linux 서버 5대
   - 도커 환경에서 실행
   - OS 정보, 패키지, 설정값 수집 확인 (화면에 로그 표시)

2. **AI 분석**
   - 수집된 정보 → Claude API 분석
   - 취약점 JSON 생성
   - 심각도 표시 (Critical/High/Medium/Low)

3. **웹 대시보드**
   - 메인: KPI 메트릭 (서버 상태, 취약점 카운트)
   - 목록: 취약점 심각도별 표시
   - 상세: 취약점 클릭 → 원인·조치방법·영향받는 서버 조회

## 🚨 주의사항

### 1. 범위 관리 (절대 규칙)
- 기능 요청이 오면 → "7/15 이후 v1.1에" 로 연기
- 지금도 시간이 부족한데, 범위 추가 = 완주 불가
- 일정이 밀리면 → 기능 제거 (추가 금지)

### 2. 기술 선택은 내일까지
- 선택 지연 = 개발 시간 손실
- 완벽한 선택보다 **빠른 선택**이 중요

### 3. CVE 매칭은 정적 데이터로 테스트
- NVD API 연동은 복잡함 (버전 비교 정확도, 캐싱 등)
- MVP에서는 고정된 CVE 목록으로 테스트
- 자동 연동은 v1.1에서

### 4. Windows는 제외 (Linux SSH만)
- WinRM 테스트 환경 구성에 오버헤드
- Linux 데모로도 충분히 임팩트 있음
- Windows 지원은 "향후 확장"으로 표시

### 5. 프로토타입 마인드셋
- "완벽함"이 아니라 "작동함"이 목표
- UI는 기능 우선, 디자인은 최소한 수준
- 코드 품질도 기본(테스트 커버리지, 리팩토링)만
