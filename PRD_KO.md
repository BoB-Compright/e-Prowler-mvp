# PRD: AI 기반 인프라(OS) 보안 점검 도구 (MVP)

**버전:** 2.0  
**작성일:** 2026-06-24  
**상태:** 개발 준비 완료  
**팀:** 개발 초심자 3명  
**일정:** 20일 (실제 가용 시간: 10~12일)

---

## 문제 정의

대규모 인프라(IDC 규모, 수백~수천 대 서버)를 관리하는 보안팀이 직면한 과제:

1. **수동 취약점 평가** — 인프라 보안 점검을 수동으로 진행하여 많은 시간과 자원 소비
2. **CVE 모니터링 병목** — 매일 업데이트되는 CVE를 전체 서버 플릿에서 추적하기 어려움
3. **대규모 운영 비효율** — 기존의 스캔-리포트 워크플로우가 IDC 규모로 확장되지 않음
4. **통합 가시성 부족** — 보안팀이 전사적 취약점을 한눈에 파악하고 CVE 영향도를 신속히 판단할 수 없음

결과: 취약점 발견 지연, 보안 위험 증가, 수동 작업 과중

---

## 핵심 네러티브

IDC 규모(수백~수천 대 서버)의 인프라 보안을 책임진 보안팀이 있습니다. 매일 발표되는 새로운 CVE를 전체 서버에 적용해야 하지만, 지금까지는 대부분 수동으로 진행해왔습니다. 서버 수가 늘어나면서 스캔과 분석에 들어가는 시간이 점점 증가하고, 취약점 발견과 대응 사이의 시간 차이가 벌어지고 있습니다. 특히 규제 당국의 컴플라이언스 요구가 강화되면서 취약점 관리의 속도와 투명성이 조직의 필수 과제가 되었습니다. 

이 도구는 OS 정보 수집을 완전히 자동화하고(Ansible/SSH), **AI(Claude)의 추론 능력**을 활용하여 수집된 정보에서 취약점을 식별하고 한국어로 조치방안을 제시합니다. 동시에 NVD CVE 데이터베이스와 자동 매칭하여 영향받는 서버를 식별합니다. 보안팀은 웹 대시보드에서 모든 서버의 취약점을 심각도와 영향 범위별로 한눈에 파악하고, 한국어 조치방안을 바탕으로 신속하게 대응할 수 있게 됩니다. **결과적으로 취약점 발견과 대응의 시간 차이를 단축하고, 수동 작업 부담을 크게 경감하며, 컴플라이언스 요구를 충족할 수 있습니다.**

### 네러티브 구성

| 요소 | 내용 | 출처 |
|------|------|------|
| **대상** | IDC 규모의 대규모 인프라를 관리하는 보안팀 (규모: 수백~수천 대 서버) | 명시 |
| **진짜 문제** | 수동 취약점 평가로 인한 시간 지연, CVE 모니터링 병목, 스캔-리포트 워크플로우가 규모에 맞지 않음, 취약점 발견 지연 | 명시 |
| **맥락·타이밍** | 서버 규모 증가로 기존 방식 비효율화 + 컴플라이언스 규제 강화로 취약점 관리의 속도·투명성 요구 증대 | 명시 + 추론 |
| **핵심 관점** | AI(Claude)의 추론 능력을 활용한 자동 분석 + 통합 가시성을 통한 신속한 의사결정 | 명시 |
| **해결 방향** | 자동화(Ansible) → AI 분석(Claude API) → CVE 매칭(NVD) → 통합 대시보드 | 명시 |
| **성공의 모습** | 취약점 발견-대응 시간 단축, 수동 작업 부담 경감, 전사적 취약점 가시성 확보, 컴플라이언스 대응 능력 강화 | 명시 + 추론 |

### 진단

**빠진 것:** 없음. 사용자 확인을 통해 맥락 보충 완료.

**약한 것:** 
- 현재 보안팀의 구체적 규모(몇 명), 현재 워크플로우 상 정확한 병목 시간(주당 X시간) 등 정량 지표 부재
- 기존 상용 도구(Nessus, Qualys 등)와의 구체적 비교 분석 없음 (비용, 접근성, 기술적 장점 등)

**요구와 문제의 간극:**
- 60개의 사용자 스토리 중 자산 관리(1~10번), 인증(11~16번)은 기본 기능이므로, 핵심 가치(AI 분석 + CVE 매칭 + 대시보드)와 명확히 구분 필요
- MVP 성공 기준이 3가지(수집, 분석, 대시보드)로 정의되어 있으므로, 나머지 스토리는 우선순위 재정의 권장

## 솔루션

담당자 노트북 기반 AI 인프라 보안 점검 도구:

1. **OS 정보 자동 수집** — Ansible/SSH를 통해 온프레미스 클라우드의 Linux 서버에서 OS, 패키지, 보안 설정 정보를 자동으로 추출
2. **AI 기반 취약점 분석** — Claude API를 활용해 수집된 정보를 분석하고 취약점을 식별하며 한국어로 조치방안 제시
3. **CVE 데이터 매칭** — 수집된 OS/패키지 정보와 NVD CVE 데이터베이스를 자동 매칭하여 영향받는 서버 식별
4. **웹 기반 대시보드** — 담당자가 노트북 웹 브라우저(http://localhost:3000)에서 스캔 실행, 취약점 조회, 필터링, 조치 가이드 확인

**배포 구조:**
- 담당자 노트북에서 Docker Desktop으로 FastAPI 백엔드 + React 프론트엔드 + SQLite DB 실행
- 온프레미스 클라우드 내부 네트워크를 통해 대상 서버(Linux)에 SSH/Ansible로 접근
- 취약점 데이터는 SQLite DB에 로컬 저장 (담당자가 정기적으로 백업)

MVP는 **"척추 한 줄" 3가지 핵심 기능**에 집중하여 20일 내 개발 초심자 팀이 접근 가능한 수준으로 검증합니다.

## 사용자 스토리

### 자산 관리

1. 보안 담당자로서, CSV 파일에 서버 IP, 호스트명, OS 타입, 계정 정보를 입력하고 시스템에 가져오고 싶습니다. (인벤토리 빠른 등록)
2. 보안 담당자로서, Excel(.xlsx) 파일로도 자산을 업로드하고 싶습니다. (기존 자산 추적 시스템과 연동)
3. 보안 담당자로서, JSON 형식의 자산 정의도 업로드하고 싶습니다. (IaC 도구와의 통합)
4. 시스템 관리자로서, 업로드 중 실패한 행이 어느 것인지, 왜 실패했는지 알고 싶습니다. (데이터 품질 개선)
5. 시스템 관리자로서, 업로드 결과 요약(성공 N개, 실패 M개)을 보고 싶습니다. (진행 상황 확인)
6. 보안 담당자로서, 대시보드에서 서버를 수동으로 추가하고 싶습니다. (파일 재업로드 없이 신속 대응)
7. 보안 담당자로서, 기존 서버의 정보(IP, 호스트명, 계정)를 수정하고 싶습니다. (인벤토리 최신 유지)
8. 보안 담당자로서, 폐기된 서버를 인벤토리에서 삭제하고 싶습니다. (정확한 자산 관리)
9. 보안 담당자로서, 모든 서버를 목록으로 보고 싶습니다. (IP, 호스트명, OS, 마지막 스캔 시간)
10. 보안 담당자로서, 각 서버의 상태(정상/경고/위험)를 한눈에 파악하고 싶습니다. (우선순위 결정)

### 계정 보안 및 인증

11. 시스템 관리자로서, 비밀번호와 SSH 개인키를 암호화된 형태(AES-256)로 저장하고 싶습니다. (저장 데이터 보호)
12. 시스템 관리자로서, 암호화 키를 환경변수(INFRA_SECURITY_MASTER_KEY)에서 로드하고 싶습니다. (키 관리 분리)
13. 보안 담당자로서, 계정 정보는 활성 스캔 중에만 복호화되고 싶습니다. (로그 노출 방지)
14. 시스템 관리자로서, Admin 사용자만 암호화된 계정 정보에 접근할 수 있게 하고 싶습니다. (접근 제어)
15. 사용자로서, 보안 로그인 페이지(사용자명/비밀번호)로 인증하고 싶습니다. (무단 접근 방지)
16. Admin으로서, 사용자 계정을 생성하고 관리하고 싶습니다. (사용자 관리)

### OS 정보 수집 (MVP 핵심)

17. 보안 담당자로서, Linux 서버 5~10대를 Ansible/SSH(포트 22)로 스캔하고 싶습니다. (기본 정보 수집)
18. 보안 담당자로서, OS 버전과 커널 버전 정보를 수집하고 싶습니다. (노후 시스템 식별)
19. 보안 담당자로서, CPU, 메모리, 디스크 정보를 수집하고 싶습니다. (용량 vs 취약점 연관 분석)
20. 보안 담당자로서, 호스트명과 IP 설정 정보를 수집하고 싶습니다. (네트워크 토폴로지 파악)
21. 보안 담당자로서, 설치된 패키지 목록과 버전을 완전히 수집하고 싶습니다. (CVE 데이터 매칭)
22. 보안 담당자로서, 주요 서비스와 실행 상태를 파악하고 싶습니다. (공격 표면 파악)
23. 보안 담당자로서, 방화벽 규칙과 상태를 수집하고 싶습니다. (네트워크 분할 확인)
24. 보안 담당자로서, SSH 설정값을 수집하고 싶습니다. (접근 제어 감사)
25. 보안 담당자로서, 사용자 권한 설정을 수집하고 싶습니다. (권한 상승 위험 식별)
26. 보안 담당자로서, SELinux 상태를 확인하고 싶습니다. (보안 모듈 설정 파악)
27. 보안 담당자로서, 비밀번호, API 키, 개인키 등이 ****로 마스킹되어 저장되고 싶습니다. (민감 정보 보호)
28. 보안 담당자로서, 최대 10대 서버를 동시에 스캔하고 싶습니다. (수집 시간 단축)
29. 보안 담당자로서, 스캔 실패 시 30초 간격으로 최대 3회 재시도하고 싶습니다. (일시적 오류 극복)
30. 보안 담당자로서, 실패한 서버는 건너뛰고 나머지는 계속 스캔되고 싶습니다. (부분 실패 허용)
31. 보안 담당자로서, 서버당 스캔이 5분 이상 걸리면 자동 중단되고 싶습니다. (무한 대기 방지)
32. 보안 담당자로서, 성공/실패한 서버 목록을 상세 로그로 보고 싶습니다. (문제 해결)

### AI 기반 취약점 분석 (MVP 핵심)

33. 보안 담당자로서, 수집된 OS 정보를 Claude API로 분석하여 취약점을 도출하고 싶습니다. (AI 이용)
34. 보안 담당자로서, 취약점을 CVSS 심각도(Critical/High/Medium/Low)로 분류하고 싶습니다. (우선순위)
35. 보안 담당자로서, AI 분석 결과에서 근본 원인을 한국어로 설명받고 싶습니다. (이해도 향상)
36. 보안 담당자로서, 조치 방안을 한국어로 제시받고 싶습니다. (실행 용이)
37. 보안 담당자로서, AI 분석 결과에 영향받는 서버 목록이 포함되고 싶습니다. (영향도 파악)
38. 보안 담당자로서, 각 취약점의 조치 우선순위 점수를 보고 싶습니다. (자원 배분)
39. 보안 담당자로서, 분석 결과에 CVE ID가 포함되고 싶습니다. (추가 조사)
40. 보안 담당자로서, 수집된 패키지가 CVE 데이터베이스와 자동 매칭되고 싶습니다. (적용 취약점 식별)

### CVE 데이터베이스 연동 (MVP 핵심)

41. 보안 담당자로서, NVD CVE API와 연동하여 최신 취약점 정보를 얻고 싶습니다. (위협 인텔리전스)
42. 보안 담당자로서, 설치된 패키지가 자동으로 CVE 데이터베이스와 매칭되고 싶습니다. (영향받는 시스템 식별)
43. 보안 담당자로서, CVE 데이터가 매일 자동 또는 수동으로 업데이트되고 싶습니다. (최신 정보 유지)
44. 보안 담당자로서, NVD API가 다운 중에도 로컬 캐시 데이터로 스캔이 작동하고 싶습니다. (장애 대응)
45. 보안 담당자로서, 각 취약점의 CVE ID, 제목, 설명, CVSS 스코어를 보고 싶습니다. (상세 정보)

### 웹 대시보드 (MVP 핵심)

46. 보안 담당자로서, 전체 서버 수를 KPI로 보고 싶습니다. (스캔 범위 파악)
47. 보안 담당자로서, 정상/경고/위험 서버 수를 KPI로 보고 싶습니다. (플릿 상태 평가)
48. 보안 담당자로서, Critical/High/Medium/Low 취약점 수를 KPI로 보고 싶습니다. (심각도 분포)
49. 보안 담당자로서, 심각도별 취약점 분포를 파이 차트로 보고 싶습니다. (시각적 이해)
50. 보안 담당자로서, 모든 스캔된 서버 목록(IP, 호스트명, OS, 마지막 스캔 시간)을 보고 싶습니다. (커버리지 확인)
51. 보안 담당자로서, 서버를 클릭하면 그 서버의 취약점 목록을 볼 수 있고 싶습니다. (상세 조회)
52. 보안 담당자로서, 취약점을 심각도별로 필터링하고 싶습니다. (긴급도 조절)
53. 보안 담당자로서, 각 취약점의 CVE ID, 제목, 심각도, CVSS 스코어를 보고 싶습니다. (분류 정보)
54. 보안 담당자로서, 취약점의 영향받는 서버 목록을 보고 싶습니다. (영향 범위)
55. 보안 담당자로서, 조치 방안을 한국어로 읽고 싶습니다. (실행 가능성)
56. 보안 담당자로서, CVE 상세 정보 링크(NVD)를 참고하고 싶습니다. (추가 자료)
57. 보안 담당자로서, 대시보드가 1분마다 자동 갱신되기를 원합니다. (최신 정보)
58. 보안 담당자로서, 수동 새로고침 버튼도 원합니다. (즉시 갱신)
59. Admin으로서, 모든 서버와 취약점을 볼 수 있기를 원합니다. (전체 가시성)
60. Viewer로서, 할당된 팀/부서의 서버만 보고 싶습니다. (권한 제어)

### 스캔 실행 및 데이터 플로우

61. 보안 담당자로서, 대시보드에서 수동으로 스캔을 시작할 수 있고 싶습니다. (제어)
62. 보안 담당자로서, 스캔 진행 상황을 실시간으로 보고 싶습니다. (모니터링)
63. 보안 담당자로서, 스캔 결과가 데이터베이스에 저장되고 싶습니다. (영구 기록)
64. 보안 담당자로서, 각 스캔에 타임스탬프가 기록되고 싶습니다. (이력 추적)

## 운영 환경

**담당자 노트북 중심 운영**
- 보안 담당자의 노트북이 중앙 운영 환경 (개발/운영 통합)
- Docker Desktop에서 모든 서비스 실행 (FastAPI 백엔드, React 프론트엔드, SQLite DB)
- 담당자가 웹 브라우저(http://localhost:3000)로 접근하여 스캔 실행 및 결과 조회
- 온프레미스 클라우드의 Linux 서버들과 SSH/Ansible로 통신

**제약사항**
- 단일 머신 환경: 담당자 노트북 다운 시 서비스 중단 (단순성 우선)
- 리소스 제약: 메모리 8GB 이상, 디스크 20GB 이상 권장
- 스캔 병렬도: 최대 5대 서버 동시 스캔 (노트북 CPU, 메모리 고려)
- 네트워크: 온프레미스 클라우드 내부 네트워크 접근 필요 (SSH 22번 포트)

## 구현 결정사항

### 기술 스택

- **백엔드:** Python + FastAPI (비동기, 병렬 처리 지원)
- **프론트엔드:** React (컴포넌트 기반, 대시보드 시각화)
- **데이터베이스:** SQLite (파일 기반, 별도 서버 불필요)
- **OS 정보 수집:** Ansible (오케스트레이션) + SSH (원격 실행)
- **AI 분석:** Claude API (지능형 취약점 분석)
- **CVE 데이터:** NVD API (공개, 무료, 포괄적)
- **암호화:** AES-256 (저장 데이터), bcrypt (비밀번호)
- **배포:** Docker Desktop (담당자 노트북에서 실행)

### 데이터베이스 스키마

핵심 엔티티:
- **자산(Assets)** — IP, 호스트명, OS, 계정(암호화), 마지막 스캔 시간
- **취약점(Vulnerabilities)** — CVE ID, 제목, 심각도, CVSS, 근본 원인(한국어), 조치(한국어), 영향 서버, 참고
- **CVE 캐시** — NVD API에서 가져온 로컬 CVE 데이터
- **사용자(Users)** — 사용자명, 해시 비밀번호, 역할(Admin/Viewer), 팀 할당
- **스캔 결과** — 타임스탐프, 자산 ID, 성공/실패, 수집 정보(JSON), 분석 결과

**데이터베이스 형식:** SQLite (파일: infra_security.db, 담당자 노트북 로컬 저장)
**백업:** 담당자가 infra_security.db를 정기적으로 외부 저장소에 백업 (USB, NAS, 클라우드 스토리지 등)

### API 명세 (FastAPI)

주요 엔드포인트:
- `POST /assets/upload` — CSV/Excel/JSON 파일 업로드, 파싱, 검증, 저장
- `GET /assets` — 모든 자산 조회 (페이지네이션)
- `POST /assets/{id}/scan` — 특정 자산 스캔 시작
- `POST /scan` — 전체 플릿 스캔 시작 (병렬 처리)
- `GET /vulnerabilities` — 취약점 목록 조회 (필터링, 페이지네이션)
- `GET /vulnerabilities/{cve_id}` — 특정 CVE 상세 정보
- `GET /dashboard/metrics` — 대시보드 KPI 메트릭
- `POST /auth/login` — 사용자 인증
- `POST /cve/update` — CVE 데이터베이스 수동 업데이트

### 아키텍처 결정

1. **병렬 OS 수집** — Python asyncio로 여러 서버 동시 수집 (최대 5개 워커, 노트북 리소스 고려)
2. **애플리케이션 레이어 암호화** — SQLite는 파일 기반이므로 앱에서 AES-256 암호화 관리
3. **Ansible 플레이북 동적 생성** — 자산 데이터에서 플레이북 자동 생성 (유연성)
4. **비동기 AI 분석** — OS 수집 완료 후 Claude API 비동기 호출 (UI 블로킹 방지)
5. **CVE 매칭 로직** — 패키지명 + 버전 범위 매칭 (예: CVE 영향 1.1.0-1.1.1g ↔ 서버 설치 1.1.1a)
6. **대시보드 실시간 업데이트** — 1분 폴링으로 새 결과 전달 (WebSocket 불필요, 단순성 우선)
7. **단일 머신 운영** — 담당자 노트북이 다운되면 서비스 중단 (고가용성 불필요, 개발/운영 일관성 우선)

### 팀 역할 분담 (권장)

- **개발자 1:** 백엔드(FastAPI) + Ansible 오케스트레이션 (OS 정보 수집) + Docker Compose 설정
- **개발자 2:** AI 분석 로직(Claude API) + CVE 데이터베이스 연동 + SQLite 스키마
- **개발자 3:** React 프론트엔드 + 대시보드 + 기본 UI + 로컬 웹 서버 연결

**운영 담당:**
- 담당자가 로컬 노트북에서 Docker Desktop 실행 및 관리
- infra_security.db 정기 백업 관리

## 테스트 결정사항

### 좋은 테스트란

- **외부 동작**을 검증하며, 구현 세부사항을 테스트하지 않음
- **통합 수준**에서 작성 — 자산에서 취약점 리포트까지 전체 데이터 플로우
- **실제 또는 현실적 테스트 데이터** 사용 (실제 Linux VM 또는 Ansible 모의 출력)
- Claude API 모의 금지 (대신 고정 응답 픽스쳐 사용)

### 모듈별 테스트 전략

1. **자산 관리 모듈**
   - CSV/Excel/JSON 파싱 (정상/비정상 데이터)
   - 계정 암호화/복호화 왕복
   - CRUD 작업 (생성, 조회, 수정, 삭제)

2. **OS 수집 모듈 (Ansible)**
   - Ansible 플레이북 생성
   - 병렬 실행 (5~10 서버)
   - 재시도 로직 (일시적 오류 시뮬레이션)
   - 타임아웃 처리 (5분 이상 중단 확인)
   - 민감 정보 마스킹 (비밀번호/키 마스킹 확인)

3. **AI 분석 모듈**
   - Claude API 호출 (샘플 OS 데이터)
   - CVSS 스코어별 분류
   - 한국어 출력 검증

4. **CVE 매칭 모듈**
   - 패키지 버전 매칭 로직 (openssl 1.1.1a vs CVE 1.1.0-1.1.1g)
   - NVD API 통합 (API 응답 모의)
   - CVE 캐시 폴백 (API 다운 시 로컬 사용)

5. **API 엔드포인트**
   - `/assets/upload` (정상/비정상 파일)
   - `/scan` (플릿 스캔 큐잉)
   - `/vulnerabilities` (필터링, 페이지네이션)
   - 인증 (정상/비정상 계정)

6. **대시보드 데이터 플로우**
   - KPI 계산 (서버 상태별 카운트, 심각도별 취약점 카운트)
   - 취약점 상세 페이지 (필수 필드 검증)
   - React 클라이언트 필터링 (Vitest/Jest)

### 테스트 접점 (선호)

**상위 수준 접점:** Ansible 실행 레이어와 Claude API에서 모의. FastAPI 엔드포인트 수준의 통합 테스트. 
이를 통해 전체 플로우(자산 → 수집 → 분석 → 저장 → API 응답)를 외부 의존성 없이 테스트.

## 추가 고려사항

1. **AI 활용 전략** — Claude API를 두 가지에 활용: (a) 수집된 OS 데이터 분석으로 잠재 취약점 식별, (b) 한국어 기술 조치방안 생성. ML 파이프라인 없이 AI의 추론 능력 활용.

2. **MVP 성공 기준** — 다음 3가지만 데모에서 보여주면 성공:
   - CSV에서 5~10대 Linux 서버 가져오기 + Ansible로 OS 정보 수집 (수집률 80% 이상)
   - Claude API로 수집 정보 분석하여 심각도 분류 + 한국어 조치방안 제시
   - 웹 대시보드에서 취약점 목록, 필터링, 심각도 차트 표시

3. **팀 역량 가정** — 개발 초심자 3명 + Python/JavaScript/React 기초 보유. Claude Code로 Ansible 플레이북 및 FastAPI 보일러플레이트 생성 활용으로 20일 내 완성.

4. **보안 태세** — 온프레미스 클라우드 내부 네트워크 전용. 담당자 노트북과 대상 서버 간 SSH(22번 포트) 통신. 암호화는 저장 데이터(AES-256) + 전송 중(SSH, HTTPS) 보호. 신뢰할 수 있는 내부 네트워크 가정. 노트북 분실 시 데이터 유출 가능하므로 정기 백업 필수.

5. **성능 기준** — 초기 목표: 5대 서버 스캔 <5분 (노트북 리소스), 분석 결과 <30초, 대시보드 로드 <2초. 최적화는 MVP 범위 밖.

6. **리스크 완화** — 최대 리스크는 Ansible 학습곡선. 대응: (a) 1시간 기초 교육, (b) Claude Code로 플레이북 자동 생성, (c) 실제 테스트 VM에서 광범위 테스트.
