# 자산 중심 UX 정합화 (홈 대시보드 + 자산 허브) — Design

**Date:** 2026-07-09
**Status:** Approved by user, ready for implementation plan
**Priority:** 7/15 데모 전 완료 목표
**Depends on:** A1(자산/프로젝트), A2(SSH 점검), C(스케줄링), D(CVE 모니터링) — 모두 main에 병합된 상태를 전제

## Background

초기 PRD는 "레포 하나 → 점검 실행 → 리포트"의 단건 점검 흐름이 중심이었다. 이후 A1(자산/프로젝트), A2(서버 SSH 점검), C(스케줄링), D(CVE 모니터링)가 추가되면서 데이터 모델은 자산 중심으로 이동했지만, 화면 구조는 초기 형태에 증축을 거듭한 상태로 남았다. 그 결과:

- **홈이 대시보드가 아님** — 첫 화면이 점검 실행 폼 하나뿐이라 자산/스케줄/CVE/최근 점검 현황을 한눈에 볼 곳이 없다 (초기 PRD의 Dashboard 컨셉 소실)
- **기능이 탭별로 단절** — 자산 ↔ 점검 ↔ 스케줄 ↔ CVE가 흩어져 있어 자산 하나를 관리하려면 여러 탭을 오가야 한다. 자산 상세에는 점검 실행 버튼조차 없다
- **진입점/용어 불일치** — 점검 시작 경로가 여러 개(홈 폼, 프로젝트 fleet scan)이고, 레포 중심 용어("레포지토리" 컬럼, "레포별 최근 점검")가 서버 자산에도 그대로 노출된다

이 문서는 **중심 축을 자산(Asset)으로 확정**하고, 홈 대시보드 신설 + 자산 상세의 허브화 + 그에 따른 네비게이션/용어 정리를 다룬다.

## 핵심 결정 (질의 결과)

| 질문 | 결정 |
|---|---|
| 중심 축 | **자산 중심** — 자산 상세가 허브, 홈은 자산 전체의 보안 현황 대시보드 |
| 개편 범위 | 홈 대시보드 + 자산 허브화 (전면 IA 재설계는 하지 않음, 데모 전 완주 우선) |
| 점검 진입점 | **자산 상세에서만** + 대시보드에서 자산으로 바로가기. "점검 실행" 탭 제거 |
| 대시보드 구성 | 지표 카드 줄 + 자산별 보안 현황 테이블 + 최근 점검 활동 피드 + 위험 CVE 섹션 (4개 전부) |
| 데이터 소싱 | **접근 A: 기존 lib 함수 직접 조합** (서버 컴포넌트, 자산당 N+1 조회 수용 — `/assets` 스케줄 배지와 동일한 기수용 트레이드오프). 신규 집계 store 함수/API 엔드포인트는 만들지 않음 |

## 네비게이션 (6탭 → 5탭)

```
[현재] 점검 실행 · 자산 · 프로젝트 · 점검 이력 · 상세 리포트 · 카탈로그
[변경] 대시보드 · 자산 · 프로젝트 · 점검 이력 · 카탈로그
```

- **"점검 실행" 탭 제거** — 홈(`/`)이 대시보드가 되고, 점검 시작은 자산 상세로 통일
- **"상세 리포트" 탭 제거** — "최신 run의 리포트로 가는 탭"이라는 어색한 개념을 제거. 리포트 화면(`/runs/[id]/report`) 자체는 그대로 두고, 점검 이력·대시보드 피드·자산 상세 이력에서 진입
- **카탈로그 탭 유지** (참조 문서 성격, 맨 끝 배치)
- `AppHeader`의 `latestRunId` prop과 관련 로직(상세 리포트 탭 전용)은 함께 제거

## 홈 대시보드 (`/`)

서버 컴포넌트. 기존 `listAssets`, `listRuns`, `getRunRiskSummary`, `overallRunOutcome`, `listCveMatches`, `getScheduleByAsset`를 직접 호출해 조합한다. 위에서 아래로:

### 1. 지표 카드 줄 (4장)
- **총 자산**: 전체 수 + "레포 n · 서버 m" 보조 표기
- **취약 자산**: 마지막 run의 `overallRunOutcome`이 `fail`인 자산 수
- **미해결 CVE**: 전 자산의 undismissed CVE 총수, Critical/High 수 강조 표기
- **활성 스케줄**: `enabled` 스케줄 수

### 2. 위험 CVE 섹션
- undismissed **Critical/High** CVE 상위 5개 (CVSS 내림차순): CVE ID · 자산명 · 패키지@버전 · CVSS 점수
- 각 행 클릭 → 해당 자산 상세(`/assets/[id]`)
- 0건이면 섹션 자체를 "위험 CVE 없음" 한 줄로 축약

### 3. 자산별 보안 현황 테이블
| 컬럼 | 내용 |
|---|---|
| 자산명 | 클릭 → 자산 상세 |
| 타입 | 레포/서버 |
| 마지막 점검 | 시각 + outcome 배지(양호/검토/취약). run이 없으면 "점검 이력 없음" |
| Critical/High | 마지막 run의 심각도 수 |
| 정기 점검 | 스케줄 배지(매일/매주/매월/—) — `/assets` 목록과 동일 표기 |
| CVE | 해당 자산 undismissed CVE 수 (서버 자산만, 레포는 —) |

### 4. 최근 점검 활동 피드
- 최근 run 8개: 자산명(또는 점검 대상 표기) · 수동/예약 배지 · 상태 · 시각
- 클릭 → 진행 중이면 `/runs/[id]`, 완료면 `/runs/[id]/report`

### 보조 액션 (우상단)
- "자산 등록" → `/assets/new`
- "로컬 이미지 재점검" → 기존 local_image 폴백 경로의 새 거처. 기존 `StartRunForm`의 local_image 모드 부분을 분리해 **대시보드 하단의 접이식(collapsed 기본) 섹션**으로 유지 (시연 안전장치이므로 제거하지 않음; 모달 등 새 UI 패턴은 도입하지 않음)

## 자산 허브 (`/assets/[id]`)

```
┌ 자산명 · 타입 배지 · 소속 프로젝트(링크, 미분류면 "미분류") ── [점검 시작] ┐
│ repo URL 또는 host:port                                                  │
├─────────────────────────────────────────────────┤
│ ① 정기 점검 — 기존 ScheduleForm 그대로                                    │
│ ② 감지된 CVE — 서버 자산만, 기존 CveList 그대로                           │
│ ③ 점검 이력 — 아래 참고                                                  │
└─────────────────────────────────────────────────┘
```

- **[점검 시작] 버튼** (신규 클라이언트 컴포넌트, 예: `StartScanButton.tsx`):
  - `POST /api/runs { assetId }` 호출 → 성공 시 `router.push(/runs/[id])`
  - 레포/서버 분기는 API가 이미 처리하므로(A2) 버튼은 자산 타입 무관 동일
  - 실행 중 중복 클릭 방지(disabled), 실패 시 한국어 에러 메시지 인라인 표시
- **점검 이력 섹션 개선**: 현재 `날짜 — status` 텍스트뿐이고 링크가 없음 → `/runs` 목록과 동일한 배지 스타일(outcome 배지 + 심각도 요약 + 수동/예약 배지)로 통일하고, 각 행 클릭 시 진행 중이면 `/runs/[id]`, 완료면 리포트로 이동
- 소속 프로젝트 표시 + `/projects/[id]` 링크 (역방향 연결)

## 용어/흐름 정리

| 위치 | 현재 | 변경 |
|---|---|---|
| `/runs` 테이블 헤더 | "레포지토리" | "점검 대상" (서버 run은 host IP가 표시됨) |
| `/runs` 부제 | "레포별 최근 점검 결과 · 심각도 요약 비교" | "자산별 최근 점검 결과 · 심각도 요약 비교" |
| `/runs` 빈 상태 문구 | "점검 실행 탭에서 레포 URL을 입력해 첫 점검을 시작하세요" | "자산 탭에서 자산을 등록해 첫 점검을 시작하세요" (점검 실행 탭 제거로 필수 수정) |
| `/runs` 필터 툴팁 | "이 레포의 이력만 보기" | "이 자산의 이력만 보기" |
| 헤더 타이틀 | "Container Security Pipeline" | "e-Prowler — 자산 보안 점검" (컨테이너 전용이 아니게 됐으므로) |

기존 `/` 홈의 파이프라인 설명 문구("GitHub 레포 → Docker 빌드 → …")는 대시보드로 교체되면서 자연히 사라진다.

## 엣지 케이스

| 상황 | 처리 |
|---|---|
| 자산이 0개 (첫 사용) | 대시보드에 빈 상태: "등록된 자산이 없습니다" + [자산 등록] 버튼 유도 |
| run이 하나도 없는 자산 | 현황 테이블에 "점검 이력 없음", 심각도 컬럼 "—" |
| 진행 중(running) run이 있는 자산에서 [점검 시작] | API가 반환하는 에러를 그대로 인라인 표시 (기존 API 동작 재사용, 새 검증 로직 추가 안 함) |
| 레포 자산의 CVE 컬럼 | CVE 수집은 서버 자산 전용(D)이므로 "—" 표시 |
| 삭제된 자산의 run (asset_id가 남은 이력) | 최근 활동 피드에서 자산명 대신 기존 `repoUrl`(점검 대상 문자열) 그대로 표시 |

## 테스트 전략

- 대시보드/자산 허브는 서버 컴포넌트 — 이 코드베이스 관례대로 페이지 테스트 없음, `tsc --noEmit` + dev 서버 curl/수동 검증이 수용 기준
- 새 lib 헬퍼가 생기는 경우에만(예: "자산별 마지막 run + outcome" 계산을 재사용 가능한 함수로 뽑는 경우) 해당 함수만 vitest 단위 테스트
- `StartScanButton`은 기존 `ScheduleForm`과 같은 급의 클라이언트 컴포넌트 — 페이지 테스트 관례 없음, 수동 검증
- 기존 테스트 회귀 없음 확인 (전체 스위트)

## 제외 범위 (이번 스코프 아님)

- 전면 IA 재설계, 프로젝트 화면 개편
- 대시보드 자동 새로고침/실시간 갱신 (새로고침으로 충분, MVP)
- 신규 집계 store 함수/집계 API 엔드포인트 (접근 A 채택)
- 모바일 반응형 대응
- 점검 이력(`/runs`)의 구조 변경 (용어만 수정)
